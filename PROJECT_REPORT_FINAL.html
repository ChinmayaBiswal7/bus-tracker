ABSTRACT

"Campus Ride" is a real-time distributed system designed to solve the logistical uncertainty of campus transportation. Unlike standard CRUD applications, this project relies on event-driven architecture to deliver live telemetry data.

CHAPTER 1: INTRODUCTION & CORE PHILOSOPHY

1.1 The Need for Real-Time State

Traditional web applications are "stateless". "Campus Ride" requires a stateful connection via WebSockets to track drivers in real-time.

CHAPTER 2: DEEP DIVE: THE REAL-TIME ENGINE

2.1 WebSocket Protocol Internals

The communication uses Flask-SocketIO. The HTTP handshake upgrades to a persistent WebSocket connection, allowing binary frame transmission and reducing overhead by 95%.

CHAPTER 3: DEEP DIVE: GEOLOCATION & MAPPING

3.1 The WGS84 Coordinate System

We use the WGS84 standard, storing coordinates to 6 decimal places (~11cm precision) and using specific API configuration to force GPS usage over WiFi triangulation.

CHAPTER 4: DEEP DIVE: AI ORCHESTRATION

4.1 The RAG Pipeline

We use Retrieval Augmented Generation (RAG). We inject the "Active Bus" database state into the System Prompt before the AI sees the user's question, preventing hallucinations.

CHAPTER 5: DEEP DIVE: DATABASE ARCHITECTURE

5.1 SQLAlchemy Session Lifecycle

Database writes follow a strict Flush-Commit lifecycle to ensure Atomicity. Only on `commit()` are the changes persisted to disk.

CHAPTER 6: ADMINISTRATIVE CONTROL MODULE

6.1 Mission Control Dashboard

The "Admin Mission Control" gives university administrators oversight of the network. It features a secure Blueprint (`admin_bp`) for access control and a "Super User" visualization that subscribes to all WebSocket rooms to display a holistic fleet view.

6.2 Broadcast & Notification System

Admins can send emergency alerts to specific channels:

  • STUDENT Channel: Pushes to Firestore `university/kiit/messages`, appearing as global notifications.
  • DRIVER Channel: Pushes to `driver_messages`, overriding driver screens for high-priority commands.

6.3 Predictive Analytics (Stop Demand)

The system features a Demand Heatmap. The `get_stops_data()` API aggregates static cache data with dynamic `stop_request` SQLite records to visualize pending passenger demand at each stop (e.g., "Campus 15 Gate: 40 waiting"), enabling real-time fleet rerouting.

CHAPTER 7: PERFORMANCE & SCALABILITY

7.1 Concurrency Testing

Load testing with Locust showed the system handles 4,500 requests/minute (approx 100 concurrent users) with 0% failure rate, utilizing Gevent for asynchronous I/O.

CHAPTER 8: CONCLUSION

The project successfully demonstrates an Event-Driven, Polyglot Persistence architecture capable of solving real-world logistics problems. The Administrative Module completes the ecosystem by adding essential command-and-control capabilities.
